## Anti-Patterns Verdict

**LLM assessment:** The reader largely avoids AI slop. It uses one decisive Route Green anchor, quieter alternatives, real local menu facts, and transparent safety/source states. The admin is the opposite: repeated tiny eyebrow labels, a six-card metric grid, broad shadowed bordered panels, over-rounding, and an oversized hero reproduce a generic dashboard template that the project's own design system explicitly rejects.

**Deterministic scan:** `detect.mjs --json frontend/src` returned zero findings. This is a useful false-negative lesson, not a clean bill of health: the scanner missed source-visible repeated eyebrows, clone cards, border-plus-wide-shadow styling, and 17–28px radii in `AdminPage.tsx` and `styles.css`.

**Visual overlays:** No reliable overlay is available. The isolated assessment browser had no browser backend, so mutable injection and screenshots were unavailable. HTTP 200 checks plus source/responsive review were used as fallback evidence.

## Overall Impression

The public reader has a strong foundation and can become excellent with one missing trust cue: it generates a restaurant-specific rationale but never shows it. The admin needs a hierarchy correction, not decorative polish: lead with what needs attention, demote inventory counts, and use the flat compact system already documented for the product.

## What's Working

- The reader presents one leading choice and two quieter alternatives instead of a marketplace-style grid.
- Date navigation, loading, stale, unavailable, retry, source, and dietary-safety states are honest and accessible.
- Finnish copy is concise and grounded in real lunch facts; the AI machinery stays backstage.

## Priority Issues

### [P1] The reader hides the actual reason behind each recommendation

**Why it matters:** Users see a rank and a long menu but not the generated sentence that answers “why this place today?” This weakens trust and wastes the most user-facing prompt output.

**Fix:** Render the first recommendation rationale directly below the restaurant name; render shorter companion rationales before menu details. Keep scores hidden.

**Suggested command:** `$impeccable clarify`

### [P1] Admin hierarchy starts with dashboard chrome rather than intervention

**Why it matters:** An operator must parse a hero, status strip, and six equal metrics before reaching calibration or failures. The screen does not answer “what needs attention now?”

**Fix:** Use a compact sans-serif operational header; combine health, current task, OpenAI availability, and latest update into a flat status region. Demote counts into a compact facts strip and put calibration immediately after status.

**Suggested command:** `$impeccable distill`

### [P1] Admin styling contradicts the committed design system

**Why it matters:** Repeated eyebrow scaffolding, identical cards, broad shadows, and excessive radii make the product look generated and reduce trust.

**Fix:** Remove repeated eyebrows; use sentence-case context only where necessary; cap surfaces at 14px, controls at 12px, remove resting shadows, and organize with full dividers and tonal surfaces.

**Suggested command:** `$impeccable quieter`

### [P2] Mobile decision density is still too high

**Why it matters:** Long menus can push the restaurant action below the first viewport, and “Tänään” is repeated in the date control.

**Fix:** Show the rationale and practical facts before a concise menu preview; move full menu detail behind a native disclosure; remove the duplicate today label while retaining date context.

**Suggested command:** `$impeccable adapt`

### [P2] Calibration is not sufficiently learnable or reassuring

**Why it matters:** Score dimensions require recall, feedback persistence is only implicit, and clearing a choice is not discoverable from the labels.

**Fix:** Add concise metric definitions, explicit saved/cleared status copy, a minimum 46px target, and clearer pressed-state text.

**Suggested command:** `$impeccable clarify`

## Persona Red Flags

**Casey, distracted mobile lunch chooser:** A long leading menu can push “Ravintolan sivu” and route handoff out of the first viewport. Duplicate “Tänään” copy adds noise at the exact moment the user wants one answer.

**Jordan, first-timer:** The ranking criteria are named, but the interface withholds the restaurant-specific rationale, so the top choice can feel arbitrary.

**Sam, accessibility-dependent user:** Reader semantics and focus treatment are strong. Admin select and feedback targets fall below the documented 46px target, and disabled states rely heavily on opacity.

**Alex, admin power operator:** Six neutral metrics and repeated panels slow scanning; calibration has no fast filter, bulk action, or explicit undo vocabulary.

- The reader correctly keeps scores hidden and dietary uncertainty visible.

## Questions to Consider

- If the admin exists for exceptions, why are six neutral counts the first thing it says?
- Can the first mobile viewport show the place, reason, price/hours, and real-world handoff without exposing the entire menu?
- What should an operator do in the next 30 seconds when collection fails, and is that action unmistakable?
