## Anti-Patterns Verdict

**LLM assessment:** The app does not look generically AI-generated. It has a recognisable local wayfinding identity, restrained colour, and an appropriately flat product vocabulary. The failure is over-curation: the interface summarizes and explains at the exact moment a hungry visitor needs the food itself.

**Deterministic scan:** `detect.mjs --json src/App.tsx` returned exit code 0 and `[]`. There were no deterministic findings or false positives. The menu-density problem is behavioral and architectural rather than a detector rule.

**Visual evidence:** The deployed mobile homepage at 390 × 844 devoted most of the first viewport to “Tänään kannattaa mennä tänne,” a recommendation rationale, and one of Pancho Villa's 13 dishes. The deployed Pancho Villa page used its first viewport for a long promotional description, address, phone, and external actions; the menu began below the fold. No detector overlay was created because the Browser evaluate surface is read-only.

## Overall Impression

The product is credible and technically disciplined, but its reading order contradicts the core job. Restaurant names and complete menus should be the default visible content; recommendation prose, marketing descriptions, source mechanics, and secondary actions should follow.

## What's Working

- The selected restaurant day already renders a complete structured menu with useful hours, price, dietary markers, and source context.
- Loading, stale, empty, and error states are honest and recoverable.
- The restrained palette, focus styles, semantic landmarks, and 46px controls provide a solid accessible foundation.

## Priority Issues

### [P1] Complete homepage menus are hidden

**Why it matters:** The rank-one card renders only one course with “Lisäksi 12 muuta,” and all 35 restaurant menus are closed `<details>` rows. Users must expand or navigate before they can compare what they would actually eat.

**Fix:** Render the complete rank-one menu and every daily restaurant menu inline. Keep the list compact by reducing metadata and decoration, not by hiding food.

**Suggested command:** `$impeccable distill`

### [P1] Restaurant pages bury the menu below promotional content

**Why it matters:** On mobile, a long imported restaurant description and external actions consume the full first viewport before today's dishes appear.

**Fix:** Lead with restaurant name, date, hours/price, and the full selected-day menu. Move description, contact details, route, website, opening hours, and provenance after the menu.

**Suggested command:** `$impeccable layout`

### [P1] The strongest homepage hierarchy is generic copy

**Why it matters:** “Tänään kannattaa mennä tänne” delays the answer and repeats what the recommendation already communicates.

**Fix:** Reduce the home heading to a compact date/menu heading and let the restaurant name plus dishes carry the decision.

**Suggested command:** `$impeccable clarify`

### [P2] Recommendation alternatives prioritize rationale over menu content

**Why it matters:** Rank, address, first dish, rationale, hours, price, and arrow create a compressed comparison block while the complete menu remains absent.

**Fix:** Use the scan order restaurant → complete menu → time/price/address; demote the rationale to a single optional line after the food.

**Suggested command:** `$impeccable distill`

### [P2] The current-day control has awkward semantics

**Why it matters:** “Tänään” looks like a button but is an `aria-disabled` no-op removed from tab order.

**Fix:** Render a plain current-state label when today is selected or use a native disabled button.

**Suggested command:** `$impeccable audit`

## Persona Red Flags

**Casey, distracted mobile user:** Cannot see a full recommendation without leaving the homepage. On the restaurant page, the menu is below a long description and external actions. The one-handed, under-a-minute task is delayed by content that does not help choose food.

**Jordan, first-time visitor:** “Päivän ykkösvalinta” looks like a finished answer, but “Ruokalistalta” exposes only a sample. Jordan can reasonably mistake one dish for the restaurant's full offering.

**Sam, accessibility-dependent visitor:** Semantic landmarks, native controls, focus styles, and live regions are strong. Comparing many collapsed menus nonvisually creates unnecessary interaction and memory load; the no-op current-day button is semantically weak.

## Minor Observations

- The 50 km catchment is too broad to imply practical proximity, but the current API has no comparable distance data; avoid claiming convenience until that exists.
- Source and allergy information is important, but its detailed explanation should follow the menu rather than interrupt it.
- Native `<details>` is accessible, but it is the wrong default for the primary content users came to read.

## Questions to Consider

- Could the homepage be successful if users never need to open a restaurant page to decide?
- If the food is the decision, which non-menu line truly earns space above it?
- Should “ykkösvalinta” mean best menu only, or does the product eventually need distance and practical fit to support the word “best”?
